What Was Missing
from the PRD

Your original documents were solid â€” but these are the critical gaps we identified and added. Each one would have caused real problems if left unaddressed.

No AI Strategy at All

The PRD mentioned OTPs must be time-bound and single-use but gave no implementation path. Storing OTPs in PostgreSQL creates race conditions and doesn't expire cleanly. Redis is the right tool.
Why it matters: Without Redis, OTP brute force attacks are possible. A bad actor could try thousands of OTP combinations on a user's account.

No Error Monitoring (Sentry)
The PRD said "proper error handling & logs required" but didn't specify how. Without Sentry, you find out about production bugs when angry users call you â€” not before.
Why it matters: A payment gateway failure or OTP service outage can happen at 2am. Sentry alerts you immediately with the exact line of code that broke.

No Email System (SendGrid)
The PRD only mentioned SMS and push notifications. But critical documents â€” GST invoices, agreement copies, monthly statements â€” need to be emailed. SMS can't carry a PDF attachment.
Why it matters: Owners need email invoices for their accounting. Tenants need email copies of agreements. This is a compliance requirement in many cases.
ğŸ“Š
Analytics
BigQuery Data Warehouse
The PRD had reports and analytics but no data infrastructure. PostgreSQL is not designed for analytical queries across millions of rows. BigQuery handles this and doubles as the foundation for AI training data.
Why it matters: In 2 years, PGfy's proprietary rental market data (search trends, pricing, occupancy patterns) will be worth more than the app itself. Start collecting it now.

Quality
No Automated Testing Plan
The PRD mentioned testing in passing. With a payment system, legal agreements, and OTP security, having zero automated tests is a ticking time bomb. One bad deploy breaks payments for all users.
Why it matters: Jest for backend + Detox for mobile means you can deploy with confidence. The smart lease engine's penalty calculation especially needs test coverage â€” it's financial logic.

The PRD went from "build" straight to "launch." Going from 0 to public with no intermediate step is extremely risky. A payment bug affecting 1000 owners on Day 1 can destroy the platform's reputation permanently.
Why it matters: Pilot with 10 trusted properties for 2 weeks first. Fix everything that breaks. Then open publicly. This single step de-risks the launch by 80%.

Non-Negotiables

Engineering Principles

Rules the entire development team must follow from Day 1. These aren't suggestions â€” violating them creates technical debt that costs 10x to fix later.

01
Backend Always Leads Frontend
No frontend work starts on a feature until the API is written, tested in Postman, and documented in Swagger. Ever.
02
Schema is Sacred
Spend extra time on database design in Phase 1. Every schema change after launch is a migration that affects live data. Get it right upfront.
03
Never Touch Production Manually
All deployments go through GitHub Actions CI/CD. No SSH into production servers. No manual SQL on the production DB.
04
Zero Hardcoded Secrets
Every API key, password, and credential lives in GCP Secret Manager. If it's in the codebase, it's a security incident waiting to happen.
05
Document As You Build
Every API endpoint gets a Swagger doc the same day it's built. Every module gets a README. Documentation done at the end never gets done.
06
Soft Deletes Only
No hard deletes on any table. Use deleted_at timestamps. Agreements, payments, and visitor logs are legally required to be permanently recoverable.
07
Test on Real Devices
React Native behaves differently on real iOS/Android vs simulators. Every feature is tested on real phones from Phase 3 onwards â€” not just emulators.
08
API Versioning Always
All routes under /api/v1/. Breaking changes get a new version, old ones stay alive. Once apps are on app stores, you cannot force everyone to update.
09
AI Calls from Backend Only ADDED
Claude API is called from the backend exclusively. Never from the mobile app directly. Keeps API keys secure, allows AI logic to update without app store releases.
10
Log Everything for AI ADDED
Every user action â€” search query, filter used, property viewed, booking abandoned â€” is structured and streamed to BigQuery. This is how PGfy's AI gets smarter over time.
